Runner Framework: Architecture Approaches

How should the framework separate "engine" from "user workflows"?

A

Plugin Architecture

Framework owns the loop. Users plug in behaviors.

How it works
FRAMEWORK (fixed)
Task Queue → Pipeline Engine → Stage Runner
State Machine → Event Bus → Persistence
Process Manager → Output Parser → UI Server
PLUGINS (user-defined)
Executors: claude-code, codex, shell-script, http-api
Hooks: post-stage-git-commit, post-pipeline-create-pr
Reporters: telegram, slack, webhook
Git/PR management = a built-in plugin called git-workflow.
Ships by default, users can disable or replace it.

Pros

  • Simple mental model — "fill in the slots"
  • Framework handles all the hard stuff (queue, resume, state)
  • Easy to share plugins between users

Cons

  • Plugin API becomes a contract you can't easily change
  • Limited flexibility — if the framework loop doesn't fit, you're stuck
  • Risk of "plugin hell" — too many extension points
B

Composable Primitives

Framework provides building blocks. Users compose their own engine.

How it works
PRIMITIVES (library)
TaskQueue — priority queue with concurrency
Pipeline — stage sequencer with retry/resume
AgentProcess — spawn + parse + timeout
StateMachine — define states + transitions
StateStore — persist + broadcast changes
EventBus — pub/sub for everything
USER CODE (compose freely)
const engine = new Pipeline([
  stage('plan', claude({ prompt: '...' })),
  stage('build', codex({ prompt: '...' })),
  stage('notify', webhook({ url: '...' })),
])
Git/PR management = a utility function you call in stage callbacks.
afterStage(gitCommit()) / afterPipeline(createPR())

Pros

  • Maximum flexibility — combine pieces however you want
  • No framework lock-in — use only what you need
  • Easier to test individual pieces

Cons

  • More work for users — they assemble the engine themselves
  • Harder to build a UI on top (no standard shape)
  • "Library fatigue" — too many choices for beginners
C

Config-Driven Engine + Escape Hatches

Declare workflows in JSON/YAML. Drop to code only when needed.

How it works
ENGINE (fixed, opinionated)
Reads pipeline definitions → runs them
Built-in executors: claude, codex, shell, http
Built-in state machine, persistence, UI
PIPELINE CONFIG (JSON)
{
  "name": "research",
  "stages": [
    { "name": "gather", "executor": "claude",
      "prompt": "Research {{task.topic}}..." },
    { "name": "analyze", "executor": "claude",
      "prompt": "Analyze: {{prev.output}}..." },
    { "name": "report", "executor": "shell",
      "cmd": "pandoc -o report.pdf" }
  ],
  "on_complete": { "action": "webhook", "url": "..." }
}
ESCAPE HATCHES (when JSON isn't enough)
Custom executor: "executor": "./my-script.js"
Stage hooks: "before": "./validate.js"
Custom state transitions: "on_fail": "./decide.js"
Git/PR management = a built-in "action" you reference in config.
"on_complete": { "action": "git-pr" } — zero code needed.

Pros

  • Lowest barrier to entry — just write JSON
  • UI can introspect pipeline definitions natively
  • Closest to what you already have (pipeline.json)
  • Escape hatches prevent hitting walls

Cons

  • Config languages always grow — risk of "YAML programming"
  • Escape hatches can become the norm if config is too limited
  • Harder to type-check and debug than real code
