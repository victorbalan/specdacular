Key Findings

  1. Live progress is the critical gap — GitHub Projects has no native support for streaming agent output or real-time status updates.
  2. GitHub Projects V2 is API-mature enough — GraphQL mutations cover field updates, item creation, and status transitions needed for task tracking.
  3. PR integration is already GitHub-native — linking tasks to pull requests requires no custom bridge; the existing data model supports it.
  4. Pluggable backend preserves optionality — an interface layer lets teams choose local-only, GitHub-only, or hybrid without code changes.
  5. Offline/latency trade-off is real — local state is instant and works offline; GitHub sync adds network dependency and eventual consistency concerns.
  6. Spec storage needs a strategy — design specs are large markdown files that don't map cleanly to GitHub Project item fields or descriptions.
