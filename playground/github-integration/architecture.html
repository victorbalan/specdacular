Architecture Approaches

Architecture Diagram

How It Works

  • All task state lives in GitHub Projects V2 + Issues
  • Runner queries GitHub API for task list, status, and specs
  • Status updates go through GraphQL mutations
  • Issue body stores spec content; comments store stage results
  • PRs auto-link to issues via "Fixes #N"

Pros & Cons

Pros

  • No local state to manage
  • Free Kanban UI via Projects
  • Built-in collaboration
  • Full audit trail on Issues
  • Native PR linking

Cons

  • Loses live progress tracking
  • 200-600ms latency per API call
  • Offline = dead
  • ID lookup complexity
  • Rate limited (5000 req/hr)

Architecture Diagram

How It Works

  • Local status.json handles live progress (fast, no API calls)
  • GitHub synced at stage boundaries (status change, stage complete)
  • Reads from local cache first, GitHub as fallback
  • Issue comments store completed stages as JSON
  • Best of both: local speed + GitHub persistence

Pros & Cons

Pros

  • Fast local reads
  • Live progress tracking works
  • Offline-capable for in-progress work
  • GitHub UI still available

Cons

  • Two sources of truth
  • Sync conflict risk
  • More implementation complexity

Code Sketch

class HybridStateManager {
  constructor(localPath, githubProject) {
    this.local = new LocalCache(localPath);
    this.github = new GitHubSync(githubProject);
  }

  updateLiveProgress(taskId, progress) {
    this.local.updateLiveProgress(taskId, progress);
  }

  async completeStage(taskId, status, summary) {
    this.local.completeStage(taskId, status, summary);
    await this.github.syncStageCompletion(taskId, status, summary);
  }

  async updateTaskStatus(taskId, newStatus) {
    this.local.updateTaskStatus(taskId, newStatus);
    await this.github.updateProjectItemStatus(taskId, newStatus);
  }
}

Architecture Diagram

How It Works

  • TaskBackend interface defines all state operations
  • LocalBackend implements current file-based storage
  • GitHubBackend implements GitHub Projects V2 + Issues storage
  • Configured per project in config.json
  • Orchestrator calls interface methods, doesn't know which backend

Pros & Cons

Pros

  • Maximum flexibility
  • Clean separation of concerns
  • Swap backend per project
  • Future backends (Linear, Jira)

Cons

  • Abstraction cost upfront
  • Lowest common denominator API
  • GitHub backend loses live progress
  • More code to write and maintain

class Orchestrator {
  constructor({ backend }) {
    this.backend = backend;
  }

  async createIdea(name, desc) {
    const task = {
      id: generateId(),
      name,
      description: desc,
      status: 'idea'
    };
    return this.backend.createTask(task);
  }
}
